# 01 — Kotlin Language Features and Idioms

**Calibration:** Principal Engineer bar — Google / Meta / Amazon  
**Focus:** Kotlin-specific constructs, how they simplify Java patterns, and where the semantics differ from what you'd expect.

## The Mental Model Shift: Kotlin is Not "Better Java"

Kotlin is designed around three principles that permeate every feature:
1. **Null safety by default** — nullability is part of the type system.
2. **Conciseness without magic** — less boilerplate via language features, not runtime tricks.
3. **Interoperability first** — runs on JVM, generates bytecode compatible with Java.

Understanding the JVM bytecode generated by Kotlin idioms is what separates senior from principal.

## 1. Data Classes

`data class` auto-generates: `equals`, `hashCode`, `toString`, `copy`, and `componentN` functions (for destructuring).

```kotlin
data class User(val id: Long, val name: String, val email: String)

// Generated methods:
val user1 = User(1, "Alice", "alice@example.com")
val user2 = user1.copy(name = "Alicia")   // creates new instance with one field changed

// Structural equality — automatically generated
user1 == User(1, "Alice", "alice@example.com")  // true (equals by components)
user1 === user2                                   // false (reference equality)

// Destructuring via componentN():
val (id, name, email) = user1
// Compiles to: val id = user1.component1(); val name = user1.component2(); ...
```

### What the Bytecode Looks Like

`data class User(val id: Long, val name: String)` generates:
- A final class (unless `open` is specified — but `open data class` is a design smell).
- `equals()`: checks all constructor properties only (not properties defined in the body).
- `hashCode()`: 31-based hash over all constructor properties.
- `copy()`: a method with all properties as default parameters, delegating to the constructor.

**Critical interview point — what `data class` does NOT copy:**

```kotlin
data class Config(val settings: MutableMap<String, String>)

val c1 = Config(mutableMapOf("key" to "val"))
val c2 = c1.copy()
c2.settings["key"] = "modified"
println(c1.settings["key"])  // "modified" — copy is SHALLOW
// c1 and c2 share the same MutableMap reference
```

`copy()` is a **shallow copy**. Mutable collections or mutable objects inside a data class are shared between the original and the copy. True immutability requires immutable types for all properties.

### Data Class vs. Java Record

| Aspect | Kotlin `data class` | Java Record |
|--------|--------------------|-----------| 
| Inheritance | Cannot extend other classes; can extend interfaces | Cannot extend classes |
| Mutability | `var` properties allowed (mutable) | All components final |
| Body | Can have extra methods and properties in body | Can have custom methods |
| `copy()` | Yes — named parameter copy | No equivalent |
| Equals scope | Constructor properties only | All components |
| JVM representation | Regular class with generated methods | Separate `record` bytecode |

## 2. Sealed Classes and `when` Expressions

Kotlin sealed classes predate Java's (Kotlin 1.0, 2016 vs. Java 17, 2021). The semantics are nearly identical, but Kotlin's `when` is more expressive:

```kotlin
sealed interface Shape
data class Circle(val radius: Double) : Shape
data class Rectangle(val width: Double, val height: Double) : Shape
data class Triangle(val base: Double, val height: Double) : Shape

// when as expression — compiler enforces exhaustiveness
fun area(shape: Shape): Double = when (shape) {
    is Circle    -> Math.PI * shape.radius * shape.radius  // smart cast inside arm
    is Rectangle -> shape.width * shape.height
    is Triangle  -> 0.5 * shape.base * shape.height
    // No else needed — sealed hierarchy is exhaustive
}
```

### `when` Without an Argument — Replaces `if-else if` Chains

```kotlin
fun classify(n: Int): String = when {
    n < 0    -> "negative"
    n == 0   -> "zero"
    n < 100  -> "small"
    else     -> "large"
}
```

### Sealed Classes vs. Sealed Interfaces

```kotlin
// Sealed interface: permitted subtypes can extend other classes
sealed interface Result<out T>
data class Success<T>(val value: T) : Result<T>
data class Failure(val error: Throwable) : Result<Nothing>
object Loading : Result<Nothing>

// Sealed class: all subtypes must be in same file/package (Kotlin 1.5+: same package)
sealed class NetworkState {
    object Connecting : NetworkState()
    data class Connected(val latencyMs: Int) : NetworkState()
    data class Error(val message: String) : NetworkState()
}
```

**`out T` (covariant):** `Result<String>` is a subtype of `Result<Any>`. Safe because `T` is only produced (returned), never consumed. Used here because `Failure` can be a `Result<Nothing>` (Nothing is a subtype of everything), making it assignable to any `Result<T>`.

## 3. Object Declarations and Companion Objects

### `object` — True Singleton at the Language Level

```kotlin
object AppConfig {
    val dbUrl: String = System.getenv("DB_URL") ?: "localhost:5432"
    val maxConnections: Int = 20

    fun isProduction(): Boolean = dbUrl.contains("prod")
}

// Usage — no instantiation
AppConfig.dbUrl
AppConfig.isProduction()
```

**JVM bytecode:** `object AppConfig` compiles to a class with a `public static final AppConfig INSTANCE` field. The instance is created when the class is first loaded (class loading is synchronized by the JVM — thread-safe by JLS guarantee). No DCLP needed.

### Companion Objects

```kotlin
class OrderService private constructor(private val repo: OrderRepository) {

    companion object {
        // Factory methods — like Java static factories
        fun create(repo: OrderRepository): OrderService = OrderService(repo)

        // Named companion objects can implement interfaces
        val DEFAULT_TIMEOUT_MS = 5000L
    }
}

// Usage:
val service = OrderService.create(repo)
OrderService.DEFAULT_TIMEOUT_MS
```

**Companion ≠ static:** A companion object is a real object instance, stored as `OrderService.Companion`. It can implement interfaces — something Java static methods cannot do:

```kotlin
interface Creator<T> {
    fun create(): T
}

class MyClass {
    companion object : Creator<MyClass> {
        override fun create() = MyClass()
    }
}

// Passes the companion object as a Creator<MyClass>
fun <T> useCreator(creator: Creator<T>): T = creator.create()
val obj = useCreator(MyClass)  // MyClass refers to its companion object
```

## 4. Extension Functions and Properties

Extensions add methods to existing classes without inheritance or decoration.

```kotlin
// Add a method to String
fun String.isPalindrome(): Boolean = this == this.reversed()
"racecar".isPalindrome()  // true

// Extension on nullable receiver
fun String?.orUnknown(): String = this ?: "unknown"
val name: String? = null
name.orUnknown()  // "unknown" — no NPE even on null

// Extension property
val String.wordCount: Int get() = this.split(" ").size
"hello world kotlin".wordCount  // 3
```

**What extensions are NOT:**

```kotlin
class User(val name: String) {
    fun greet() = "Hello, $name"
}

fun User.greet() = "Hi, $name"  // Extension — does NOT override the member!

val user = User("Alice")
user.greet()  // "Hello, Alice" — member always wins over extension
```

Extensions are **statically dispatched** — the extension function to call is determined at compile time by the declared type, not the runtime type:

```kotlin
open class View
class Button : View()

fun View.click() = println("View clicked")
fun Button.click() = println("Button clicked")

val v: View = Button()
v.click()  // "View clicked" — static dispatch on declared type View
```

**JVM bytecode:** Extension functions compile to **static methods** with the receiver as the first parameter: `StringKt.isPalindrome(String receiver)`.

## 5. Null Safety — The Type System Contract

Kotlin's type system has two kinds of types: `T` (non-nullable) and `T?` (nullable). The compiler enforces null safety at compile time.

### Safe Call `?.` and Elvis `?:`

```kotlin
val user: User? = findUser(id)

// Safe call — returns null if user is null
val email: String? = user?.email

// Elvis — provide default if null
val email: String = user?.email ?: "no-email@example.com"

// Chained safe calls
val cityLength: Int? = user?.address?.city?.length

// Elvis with throw
val city: String = user?.address?.city
    ?: throw IllegalStateException("User has no city")
```

### `let` for Null-Safe Blocks

```kotlin
val user: User? = findUser(id)

// Only executes the block if user is non-null
user?.let { u ->
    sendEmail(u.email)
    updateLastLogin(u.id)
}

// With it (implicit receiver)
user?.let { sendEmail(it.email) }
```

### `!!` — The Checked Assertion

```kotlin
val user: User? = findUser(id)
val name: String = user!!.name  // NullPointerException if user is null
```

`!!` is a code smell. It tells the compiler "I know this is non-null" but bypasses type safety. Legitimate uses: interoperating with Java APIs that return nullable types you KNOW are non-null (and can document why). In Kotlin-only code, prefer restructuring to avoid `!!`.

### Null Safety with Generics

```kotlin
fun <T> List<T>.firstOrNull(): T? = if (isEmpty()) null else this[0]

// T? is different from T when T is already nullable
// List<String?>.firstOrNull() returns String?? — which Kotlin collapses to String?
// This is called "platform types" when interoperating with Java
```

### Platform Types (Java Interop)

When calling Java APIs that return a non-annotated type, Kotlin gives you a **platform type** `T!` — it could be null or non-null. The compiler doesn't enforce null checks.

```kotlin
// Java:
public String findName() { return null; }  // returns null but not @Nullable annotated

// Kotlin call:
val name: String = JavaClass().findName()  // compiles — platform type String!
// NullPointerException at runtime when name is used — Kotlin trusted you
```

**Fix:** Annotate Java APIs with `@Nullable`/`@NonNull` (JSR-305). Or wrap the call: `val name: String? = JavaClass().findName()` (safe).

## 6. Named and Default Parameters — Eliminating the Builder Pattern

```kotlin
// In Java: need a Builder class for a constructor with many parameters
// In Kotlin: named and default parameters handle this at the language level

data class EmailConfig(
    val host: String,
    val port: Int = 587,
    val useTls: Boolean = true,
    val timeout: Duration = Duration.ofSeconds(30),
    val username: String? = null,
    val password: String? = null
)

// Creation — only specify what differs from defaults
val config = EmailConfig(host = "smtp.example.com", username = "me@example.com")
val devConfig = EmailConfig(host = "localhost", port = 1025, useTls = false)

// Copy with modification
val prodConfig = devConfig.copy(host = "smtp.prod.com", port = 587, useTls = true)
```

**Why this eliminates the Builder pattern:** The Builder pattern exists in Java because constructors can't have default or named arguments. Kotlin's default parameters + named arguments + `copy()` provides all Builder benefits without the ceremony.

**The `@JvmOverloads` bridge for Java interop:**

```kotlin
// Kotlin function with defaults — Java callers can't use defaults
fun createUser(name: String, role: String = "USER", active: Boolean = true): User

// @JvmOverloads generates Java overloads: (name), (name, role), (name, role, active)
@JvmOverloads
fun createUser(name: String, role: String = "USER", active: Boolean = true): User
```

## 7. Delegation — The `by` Keyword

### Interface Delegation

```kotlin
interface Logger {
    fun log(message: String)
    fun warn(message: String)
}

class ConsoleLogger : Logger {
    override fun log(message: String) = println("[LOG] $message")
    override fun warn(message: String) = println("[WARN] $message")
}

// AuditLogger delegates everything to ConsoleLogger, adds audit
class AuditLogger(private val delegate: Logger = ConsoleLogger()) : Logger by delegate {
    override fun log(message: String) {
        auditService.record(message)   // extra behavior
        delegate.log(message)          // delegate to ConsoleLogger
    }
    // warn() is automatically delegated — no override needed
}
```

**JVM bytecode:** The compiler generates bridge methods in `AuditLogger` that simply call the same method on `delegate`. This is the Decorator pattern with zero boilerplate.

### Property Delegation

```kotlin
class AppConfig {
    // Delegates to a lazy initializer — computed on first access, cached
    val heavyResource: HeavyResource by lazy { HeavyResource.init() }

    // Delegates to a map — config-map backed properties
    val map: Map<String, Any> = loadConfig()
    val host: String by map
    val port: Int by map

    // Observable property — fires on change
    var count: Int by Delegates.observable(0) { property, old, new ->
        println("${property.name} changed from $old to $new")
    }

    // Vetoable — can reject the change
    var positiveOnly: Int by Delegates.vetoable(0) { _, _, new -> new >= 0 }
}
```

**`by lazy` details:**
- Default mode: `LazyThreadSafetyMode.SYNCHRONIZED` — thread-safe, uses `synchronized`.
- `LazyThreadSafetyMode.PUBLICATION` — allows multiple threads to initialize, uses `volatile`, first-write wins.
- `LazyThreadSafetyMode.NONE` — no synchronization, fastest, only safe for single-threaded use.

```kotlin
// For virtual-thread-heavy code (Java 21+): avoid SYNCHRONIZED (it pins)
val resource by lazy(LazyThreadSafetyMode.PUBLICATION) { expensiveInit() }
```

## 8. Inline Functions and Reified Generics

### `inline` Functions

```kotlin
// Without inline: lambda is an object allocation + virtual dispatch
fun measureTime(action: () -> Unit): Long {
    val start = System.nanoTime()
    action()
    return System.nanoTime() - start
}

// With inline: lambda body is inlined at call site — no object, no virtual call
inline fun measureTime(action: () -> Unit): Long {
    val start = System.nanoTime()
    action()           // this line is replaced by the lambda body at call site
    return System.nanoTime() - start
}
```

**When `inline` matters:** High-frequency utility functions that take lambdas (logging, timing, resource management). Each non-inline lambda call allocates a `Function0` object — with millions of calls per second, this is measurable GC pressure.

### `reified` Type Parameters

Normal Kotlin generics are erased (same as Java). `inline` functions with `reified` type parameters preserve the type at the call site:

```kotlin
// Cannot do this with regular generics (T is erased)
fun <T> parseJson(json: String): T {
    return gson.fromJson(json, T::class.java)  // COMPILE ERROR — T is not known
}

// With reified: T is available at the call site (inlined there)
inline fun <reified T> parseJson(json: String): T {
    return gson.fromJson(json, T::class.java)  // T::class.java is substituted at call site
}

// Usage:
val user: User = parseJson<User>(jsonString)
val users: List<User> = parseJson<List<User>>(jsonArray)  // works! TypeToken not needed
```

**How it works:** Since the function is inlined at the call site, the compiler substitutes the actual type for `T` in the inlined code. The bytecode at the call site contains `User::class.java` — not a type variable.

**Limitation:** `reified` only works in `inline` functions — the type must be known at the call site.

## 9. Value Classes (Inline Classes)

```kotlin
@JvmInline
value class UserId(val value: Long)

@JvmInline
value class EmailAddress(val value: String) {
    init {
        require(value.contains("@")) { "Invalid email: $value" }
    }

    fun domain(): String = value.substringAfter("@")
}
```

**At runtime:** When used in non-generic contexts, a `value class` is represented as its underlying type — no object allocation:

```kotlin
fun sendEmail(to: EmailAddress) { ... }

// Calling: sendEmail(EmailAddress("alice@example.com"))
// Compiled to: sendEmail("alice@example.com")  — String in bytecode, no wrapper object
```

**Why use value classes:** Type-safe identifiers that have zero runtime cost. `OrderId(42L)` and `UserId(42L)` are different types at compile time, same `Long` at runtime. Prevents `fun process(userId: Long, orderId: Long)` confusion where arguments can be swapped silently.

**Limitations:**
- Must have exactly one property.
- Cannot participate in inheritance (cannot extend classes, can implement interfaces).
- In generic contexts (`List<UserId>`), boxing occurs — `UserId` is heap-allocated.

## 10. Smart Casts

```kotlin
fun describe(obj: Any) {
    if (obj is String) {
        // Smart cast: obj is automatically String inside this branch
        println(obj.length)    // no explicit (obj as String) needed
    }

    when (obj) {
        is Int    -> println("int: ${obj * 2}")   // smart cast to Int
        is String -> println("string: ${obj.uppercase()}")  // smart cast to String
        is List<*> -> println("list of ${obj.size}")         // smart cast to List<*>
    }
}
```

**Smart cast conditions — when the compiler WILL NOT smart cast:**

```kotlin
class Container(var value: Any?)  // var, not val

val c = Container("hello")
if (c.value is String) {
    println(c.value.length)  // COMPILE ERROR — c.value is a var property
    // Another thread could change c.value between the check and the use
}

// Fix: capture in a local variable (local vals are stable)
val v = c.value
if (v is String) {
    println(v.length)  // OK — v is a local val, cannot change
}
```

## Interview Q&A

### Q1 `[Principal]` Kotlin `data class` generates `equals` based on constructor properties only. What is the production bug this causes with inheritance?

**Answer:**

```kotlin
open class Animal(val name: String)
data class Dog(val breed: String, override val name: String) : Animal(name)

val d1 = Dog("Labrador", "Buddy")
val d2 = Dog("Labrador", "Buddy")

d1 == d2  // true — data class equals uses Dog's constructor properties (breed, name)

// The bug: equals is symmetric for Dog==Dog but breaks with subclasses
data class Poodle(override val name: String) : Dog("Poodle", name)
val p = Poodle("Buddy")
println(d1 == p)   // false — different classes
println(p == d1)   // false — symmetric, but it's confusing that Dog("Poodle","Buddy") != Poodle("Buddy")
```

More critically: `data class` generates `equals` that checks `other is T` (exact type, or more precisely uses `javaClass == other?.javaClass`). If `Dog` and `Poodle` are in different inheritance levels, equality is non-transitive — violating the equals contract.

**Rule:** Never use `data class` with inheritance (non-data parent classes). Use sealed class hierarchies instead, where each leaf is a `data class` and the sealed parent is never directly instantiated.

### Q2 `[Principal]` What is the difference between `object` and `companion object`, and when does a companion object lose its value over a Java static method?

**Answer:**

`object SomeName` is a standalone singleton — declared at the top level or nested inside a class, but unrelated to any particular class instance.

`companion object` is tied to a specific class — it's the class's "static namespace." There is exactly one companion object per class.

```kotlin
class PaymentService {
    companion object {
        fun create(): PaymentService = PaymentService()
    }
}
PaymentService.create()  // looks like a static call in Kotlin

// In Java interop:
PaymentService.Companion.create()  // or PaymentService.create() with @JvmStatic
```

**Where companion objects add value over Java statics:**
1. **Can implement interfaces:** `companion object : Factory<PaymentService>` — statics can't.
2. **Can be passed as objects:** `listOf(PaymentService, OrderService).forEach { it.create() }` where both companions implement `Factory<*>`.
3. **Extension functions on companion objects:** `fun PaymentService.Companion.fromConfig(c: Config) = ...`

**Where companion objects are worse:**
- `@JvmStatic` annotation needed for Java callers to use the clean `PaymentService.create()` syntax — without it, Java must call `PaymentService.Companion.create()`.
- Companion state is shared globally — same thread-safety concerns as Java statics.

### Q3 `[Principal]` Extension functions are statically dispatched. What production bug does this cause and when should you use a member function instead?

**Answer:**

Static dispatch means: the compiler decides which extension to call based on the **declared (static) type**, not the runtime type. This breaks polymorphism.

```kotlin
open class Shape
class Circle : Shape()

fun Shape.draw() = println("Drawing shape")
fun Circle.draw() = println("Drawing circle")

fun render(shapes: List<Shape>) {
    shapes.forEach { it.draw() }  // always "Drawing shape" — static dispatch on Shape
}

render(listOf(Circle()))  // prints "Drawing shape" — NOT "Drawing circle"
```

**When this causes a production bug:** You add an extension function intending it to be "overridden" for subtypes. It won't be. All subtypes get the same extension.

**Rule:** Use a **member function** whenever you need polymorphic dispatch (runtime type determines which method runs). Use **extension functions** for utility behavior that does not vary by subtype — e.g., `String.isPalindrome()`, `List<T>.second()`.

A good heuristic: if the function needs access to `private` or `protected` members → member function. If it's general utility operating only on the public API → extension function.

### Q4 `[Principal]` Explain `by lazy` thread safety modes. When would you use `PUBLICATION` over `SYNCHRONIZED`, and what is the Java 21 virtual thread implication?

**Answer:**

**`SYNCHRONIZED` (default):** A `synchronized` block around the initialization. Only one thread initializes; others block. Safe. On Java 21 virtual threads: `synchronized` pins the carrier thread (see `03-concurrency-and-loom.md`). If the lazy initializer does I/O or is slow, carrier threads are pinned — a serious performance problem with thousands of virtual threads.

**`PUBLICATION`:** Uses `@Volatile` on the backing field. Multiple threads may run the initializer concurrently. The first to write via a volatile write "wins" — subsequent writes are visible and the value becomes stable. Thread-safe because the volatile write ensures the initialized value is flushed to main memory before being read.

```kotlin
// Safe for virtual threads — no synchronized block, no carrier pinning
val cache by lazy(LazyThreadSafetyMode.PUBLICATION) {
    databaseCall()  // blocking I/O — OK, no carrier pinned
}
```

**Trade-off:** `PUBLICATION` may run the initializer multiple times (multiple threads race). Only choose it when:
1. Initialization is idempotent (running twice is safe).
2. Initialization is expensive and you want to avoid the `synchronized` lock contention.

**`NONE`:** No synchronization. Fastest. Only safe when access is single-threaded or otherwise externally synchronized.
