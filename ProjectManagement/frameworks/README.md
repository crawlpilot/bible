## When to Use Each

| Situation | Framework |
|-----------|-----------|
| "What should the team focus on this quarter?" | OKRs |
| "How does this team's work connect to company strategy?" | V2MOM → OKR cascade |
| "Which of these 20 features do we build first?" | RICE |
| "I'm drowning in requests — how do I prioritize?" | Eisenhower |
| "We have conflicting priorities from two VPs" | RICE + OKR alignment check |

---

## Principal Engineer Usage

These frameworks are tools for influence, not bureaucracy. A principal engineer uses them to:
- **Make trade-offs explicit** — RICE surfaces why Feature A beats Feature B without politics
- **Align without authority** — OKRs create shared direction across teams you don't manage
- **Say no credibly** — "That's a low-RICE item" or "It doesn't map to our current OKRs" is more powerful than "we don't have time"
- **Communicate up** — OKRs translate engineering work into language executives track
