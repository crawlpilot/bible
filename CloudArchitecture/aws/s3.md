# AWS S3 (Simple Storage Service)

## Overview
S3 is AWS's foundational object storage service — 11 nines (99.999999999%) durability, effectively unlimited scale, 5,500 GET/s and 3,500 PUT/s per prefix per second, and a global presence. It is the data lake backbone, the artifact store, the static web host, and the durable persistence layer for virtually every AWS data workload.

**Not a filesystem**: S3 is a key-value store. "Folders" are prefixes in object keys. There is no rename (copy + delete), no partial update, and no POSIX file locks.

## Storage Classes

| Class | Durability | Availability | Min duration | Cost (storage) | Use case |
|---|---|---|---|---|---|
| **S3 Standard** | 11 9s | 99.99% | None | $0.023/GB | Active data, frequent access |
| **S3 Intelligent-Tiering** | 11 9s | 99.9%–99.99% | None | $0.023+$0.0025/1K monitored | Unknown access patterns |
| **S3 Standard-IA** | 11 9s | 99.9% | 30 days | $0.0125/GB + retrieval fee | Infrequent but instant access |
| **S3 One Zone-IA** | 11 9s (one AZ) | 99.5% | 30 days | $0.01/GB | Reproducible data, IA |
| **S3 Glacier Instant** | 11 9s | 99.9% | 90 days | $0.004/GB | Archive, ms retrieval |
| **S3 Glacier Flexible** | 11 9s | 99.99% | 90 days | $0.0036/GB | Archive, minutes-hours retrieval |
| **S3 Glacier Deep Archive** | 11 9s | 99.99% | 180 days | $0.00099/GB | Compliance archive, 12hr retrieval |

**Intelligent-Tiering** moves objects automatically between access tiers. Use it for data lakes where access patterns are unpredictable. No retrieval fees, only monitoring fee ($0.0025/1,000 objects/month).

## S3 Lifecycle Policies

Automate storage class transitions and expiration:

```json
{
  "Rules": [{
    "ID": "data-lake-lifecycle",
    "Filter": {"Prefix": "raw/"},
    "Status": "Enabled",
    "Transitions": [
      {"Days": 30, "StorageClass": "STANDARD_IA"},
      {"Days": 90, "StorageClass": "GLACIER_IR"},
      {"Days": 365, "StorageClass": "DEEP_ARCHIVE"}
    ],
    "Expiration": {"Days": 2555},
    "AbortIncompleteMultipartUpload": {"DaysAfterInitiation": 7}
  }]
}
```

**Always add `AbortIncompleteMultipartUpload`** — incomplete multipart uploads are invisible but still billed. 7-day abort is standard.

## Performance & Request Limits

**Per prefix limits** (a prefix is everything before the last `/` in a key):
- **3,500 PUT/COPY/POST/DELETE requests/s** per prefix
- **5,500 GET/HEAD requests/s** per prefix

**Scaling pattern**: For >5,500 GET/s, distribute objects across multiple prefixes:
```
# Single prefix — bottlenecked at 5,500 GET/s:
s3://bucket/data/2024/01/15/object1.parquet

# Sharded across 10 prefixes — 55,000 GET/s possible:
s3://bucket/data/shard-0/2024/01/15/object1.parquet
s3://bucket/data/shard-1/2024/01/15/object2.parquet
```

**Multipart Upload**: Required for objects > 5GB; recommended for > 100MB. Enables parallel upload, resume on failure.
```
Recommended part size: 8–16 MB (balance between part count and overhead)
Max parts: 10,000
Max object size: 5 TB
```

**S3 Transfer Acceleration**: Uses CloudFront edge network for uploads from distant clients. Up to 500% faster for intercontinental uploads. Cost: +$0.04–0.08/GB.

**S3 Byte-Range Fetches**: Download only the bytes you need. Critical for Parquet column-store reads — fetch only the row groups and column chunks you need, not the entire file. Reduces Athena scan costs dramatically.

## S3 Security Model

**Access control layers** (evaluated in order):
1. **IAM policy** (identity-based): what can this principal do across all S3?
2. **S3 bucket policy** (resource-based): who can access this bucket?
3. **S3 ACLs** (legacy): avoid for new architectures; bucket/object-level grants
4. **Block Public Access** settings: account-level override that blocks all public access regardless of bucket policy

```json
// Secure bucket policy: enforce HTTPS only
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyNonTLS",
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:*",
    "Resource": ["arn:aws:s3:::my-bucket", "arn:aws:s3:::my-bucket/*"],
    "Condition": {"Bool": {"aws:SecureTransport": "false"}}
  }]
}
```

**Critical security settings**:
- **Block Public Access**: enable at account level and bucket level — prevents accidental public exposure
- **Enforce HTTPS**: bucket policy denying `aws:SecureTransport: false`
- **Server-Side Encryption**: SSE-S3 (free, AWS-managed), SSE-KMS (audit trail, key rotation, costs per API call), SSE-C (customer key)
- **S3 Object Lock**: WORM (write-once-read-many) compliance; Governance mode (admin can override) vs Compliance mode (no one can delete until retention expires)
- **MFA Delete**: requires MFA for object deletion (versioned buckets)
- **VPC Endpoint (Gateway)**: restrict bucket access to only VPC traffic via bucket policy `aws:sourceVpce`

## S3 Event Notifications

Trigger processing when objects are created/deleted/restored:

| Destination | Use case | Latency |
|---|---|---|
| SQS | Queue-based processing with retry | Seconds |
| SNS | Fan-out to multiple processors | Seconds |
| Lambda | Direct serverless processing | Seconds |
| EventBridge | Rule-based routing; schema registry; replay | ~500ms |

**EventBridge vs native S3 notifications**: Use EventBridge for complex routing, replay, cross-account delivery. Use native S3 notifications for simple, high-throughput Lambda triggers.

## S3 as Data Lake

**Partitioning strategy for Athena/Glue query performance**:
```
s3://datalake/events/
  year=2024/month=01/day=15/hour=14/
    part-00000.parquet
    part-00001.parquet
```
Hive-style partitioning (`key=value`) is auto-recognised by Glue/Athena. Query with `WHERE year='2024' AND month='01'` prunes partitions — reads only relevant files.

**File format recommendations**:
| Format | Best for | Why |
|---|---|---|
| Parquet | Analytics, Athena | Columnar; predicate pushdown; 3–10× smaller than CSV |
| ORC | Hive, EMR heavy workloads | Columnar; better compression than Parquet for Hive |
| Avro | Event streaming, schema evolution | Row-based; good for full-row reads; schema embedded |
| JSON | Raw ingestion only | Human-readable; 5–10× larger than Parquet for analytics |
| CSV | Import/export only | Least efficient; no schema; no compression |

**File size optimisation** for Athena:
- Files < 128MB: too many small files → high metadata overhead, slow listing
- Target: 128MB–1GB per file after compression
- Use S3 Inventory + S3 Batch Operations to compact small files periodically, or Glue Compaction job

## S3 Versioning & Replication

**Versioning**: Keeps all versions of every object. Protects against accidental delete and overwrite. When combined with Object Lock, provides compliance-grade immutability.
- Cost: storage for every version; lifecycle rules should expire old versions

**Cross-Region Replication (CRR)**: Asynchronously replicate to another region.
- Latency: typically < 15 minutes (monitored via S3 Replication Time Control with SLA)
- Use for: DR, data sovereignty, latency reduction for multi-region reads

**Same-Region Replication (SRR)**: Replicate within same region (different account or same account different bucket). Use for: account separation, log aggregation, test/prod data sync.

## S3 Presigned URLs

Allow time-limited access to private objects without sharing credentials:
```python
url = s3.generate_presigned_url(
    'get_object',
    Params={'Bucket': 'my-bucket', 'Key': 'private/report.pdf'},
    ExpiresIn=3600  # 1 hour
)
```
Use for: file download links in web apps, upload directly from browser to S3 (PUT presigned URL bypasses your server), temporary file sharing.

**Presigned POST**: More controlled uploads — specify content-type, size limits, and prefix in the policy. Prevents users from uploading to arbitrary keys.

## Monitoring & Alerting

| Metric | Source | Alert condition |
|---|---|---|
| `4xxErrors` | S3 request metrics | Spike → misconfigured access policies or client bugs |
| `5xxErrors` | S3 request metrics | > 0 → S3 service issue |
| `BucketSizeBytes` | CloudWatch daily metric | Sudden spike → unexpected data growth |
| `NumberOfObjects` | CloudWatch daily metric | Very high (>1M) in hot prefix → performance impact |
| S3 Storage Lens | S3 console | Incomplete MPU, unencrypted buckets, no versioning |
| CloudTrail S3 events | CloudTrail | `DeleteObject` on production bucket → immediate alert |
| S3 Access Logs | S3 access logging | Unusual GET patterns → potential data exfiltration |

Enable **S3 Storage Lens** for org-wide visibility into storage usage, cost, and security posture.

## Best Practices

1. **Block Public Access at account level** — never rely on bucket-level only; one misconfigured bucket = data breach
2. **Enable versioning on all production buckets** — protection against accidental delete; required for Object Lock
3. **Use SSE-KMS for sensitive data** — audit trail of all decryption events; key policy controls access
4. **Lifecycle rules on all buckets** — including `AbortIncompleteMultipartUpload`; otherwise incomplete MPU charges accumulate silently
5. **Use S3 Gateway Endpoint** — eliminates NAT Gateway costs for S3 access from VPC; free
6. **Use Intelligent-Tiering for data lakes** — unknown access patterns; no retrieval fee
7. **Use Parquet + partitioning for analytics data** — 10–100× query cost reduction in Athena
8. **Enforce HTTPS via bucket policy** — deny all HTTP requests
9. **Use CloudTrail data events** for S3 — log all GetObject/PutObject on sensitive buckets
10. **Separate buckets by classification** — public assets, logs, production data, PII data in separate buckets with different policies

## FAANG Interview Points

**"How do you design a data lake on S3?"**: S3 with Hive-partitioned Parquet files → Glue Catalog → Athena for ad-hoc SQL. Raw zone (JSON, as-is) → curated zone (Parquet, cleaned) → aggregated zone (pre-computed metrics). Lifecycle rules move cold data to Glacier.

**"How does S3 achieve 11 nines durability?"**: Objects are stored redundantly across a minimum of 3 AZs within a region, with multiple copies per AZ. Durability is maintained via checksums and automatic repair.

**"Design file upload for 1M users uploading 10MB images"**: Browser → presigned PUT URL (generated by API) → direct S3 upload (bypass your servers entirely) → S3 event notification → Lambda for processing (resize, metadata extract). Never route file bytes through your application servers.

**"S3 vs EFS vs EBS"**: S3 = object store, unlimited scale, no filesystem semantics, for data lakes, backups, artifacts, static assets. EFS = POSIX filesystem, shared across instances, for shared config, CMS, home directories. EBS = block storage, attached to one EC2 instance, for databases, OS volumes.
