# Application Layer — CQRS, Commands, Queries & Use Case Orchestration

The application layer is the **use case orchestrator**. It:
- Receives commands/queries from the API layer
- Loads aggregates from repositories
- Delegates all business logic to the domain layer
- Publishes domain events after transaction commits
- Has NO business logic of its own — thin by design

---

## CQRS Split

The application layer is split into two stacks:

```
Commands (writes):
  API → CommandHandler → loads Aggregate → calls domain method → saves → publishes events

Queries (reads):
  API → QueryHandler → queries read model (optimized projection) → returns DTO

Why split?
  Writes go through the rich domain model (invariants enforced).
  Reads bypass the domain model entirely — query a denormalized projection optimized for the screen.
  This avoids the N+1 problem and loading unnecessary aggregate state for simple read operations.
```

Commands are **immutable value objects** representing user intent. Named in imperative ("initiate", "confirm") not passive ("payment initiation request").

## Command Handlers

Each use case has exactly one command handler. Handlers are application-layer classes with no domain logic — they orchestrate.

### InitiateUpiPaymentCommandHandler

```java
package com.paytm.payment.application.handler;

import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class InitiateUpiPaymentCommandHandler {

    private final PaymentRepository paymentRepository;
    private final WalletRepository walletRepository;          // cross-BC, via ACL if needed
    private final UserKycProfileService kycService;           // reads from Identity BC cache
    private final FraudServiceACL fraudServiceACL;
    private final NpciUpiAdapter npciAdapter;
    private final DomainEventPublisher eventPublisher;
    private final PaymentLimitService limitService;

    public InitiateUpiPaymentCommandHandler(
        PaymentRepository paymentRepository,
        WalletRepository walletRepository,
        UserKycProfileService kycService,
        FraudServiceACL fraudServiceACL,
        NpciUpiAdapter npciAdapter,
        DomainEventPublisher eventPublisher,
        PaymentLimitService limitService
    ) {
        this.paymentRepository = paymentRepository;
        this.walletRepository = walletRepository;
        this.kycService = kycService;
        this.fraudServiceACL = fraudServiceACL;
        this.npciAdapter = npciAdapter;
        this.eventPublisher = eventPublisher;
        this.limitService = limitService;
    }

    /**
     * Orchestration steps for initiating a UPI payment:
     * 1. Idempotency check — prevent duplicate payments
     * 2. KYC & limit check — within daily limits?
     * 3. Fraud risk check — should we allow this payment?
     * 4. Create payment aggregate — initiate
     * 5. If risk approved: call NPCI
     * 6. Save payment + publish domain events
     * 7. Return result to API layer
     *
     * @Transactional: the DB save and outbox write are atomic.
     * Kafka publishing happens AFTER the transaction commits (outbox pattern).
     */
    @Transactional
    public InitiatePaymentResult handle(InitiateUpiPaymentCommand command) {

        // ── Step 1: Idempotency ─────────────────────────────────────────
        PaymentReferenceId referenceId = PaymentReferenceId.of(command.clientIdempotencyKey());
        Optional<Payment> existingPayment = paymentRepository.findByReferenceId(referenceId);
        if (existingPayment.isPresent()) {
            // This is a retry — return the existing payment's state
            Payment existing = existingPayment.get();
            return new InitiatePaymentResult(
                existing.getId().getValue(),
                existing.getStatus().name(),
                true  // isRetry = true
            );
        }

        // ── Step 2: Load KYC Profile & Check Limits ─────────────────────
        UserId payerId = UserId.of(command.payerUserId());
        UserKycProfile kycProfile = kycService.getProfile(payerId);

        // Sum of today's completed UPI payments for this user
        Money todayUpiUsage = paymentRepository
            .findCompletedPaymentsForUserOnDate(payerId, LocalDate.now())
            .stream()
            .filter(Payment::isUpiPayment)
            .map(Payment::getAmount)
            .reduce(Money.zero("INR"), Money::add);

        Money requestedAmount = Money.ofInr(command.amount());

        // Domain service enforces RBI limits + KYC limits
        limitService.assertWithinUpiLimits(requestedAmount, todayUpiUsage, kycProfile);

        // ── Step 3: Fraud Risk Evaluation ───────────────────────────────
        // Synchronous pre-payment check: ~50ms SLA from fraud service
        RiskDecision riskDecision = fraudServiceACL.evaluateUpiRisk(
            payerId,
            Vpa.of(command.payerVpa()),
            Vpa.of(command.payeeVpa()),
            requestedAmount,
            DeviceFingerprint.of(command.deviceFingerprint())
        );

        // ── Step 4: Create Payment Aggregate ────────────────────────────
        Payment payment = Payment.initiate(
            referenceId,
            requestedAmount,
            new UpiPaymentMethod(
                Vpa.of(command.payerVpa()),
                Vpa.of(command.payeeVpa()),
                command.note()
            ),
            new PaymentParticipant(payerId, kycProfile.displayName(), command.payerVpa()),
            new PaymentParticipant(
                resolvePayeeUserId(command.payeeVpa()),
                command.payeeDisplayName(),
                command.payeeVpa()
            )
        );

        // ── Step 5: Apply Risk Decision ─────────────────────────────────
        payment.startRiskCheck();

        if (riskDecision.isApproved()) {
            payment.approveRisk(riskDecision);
            // Payment is now in PROCESSING state — send to NPCI
        } else if (riskDecision.requiresChallenge()) {
            // MFA challenge required — save and return challenge response
            // (2FA flow: PIN, biometric, OTP)
            paymentRepository.save(payment);
            return new InitiatePaymentResult(
                payment.getId().getValue(),
                "CHALLENGE_REQUIRED",
                false
            );
        } else {
            payment.blockForRisk(riskDecision);
            paymentRepository.save(payment);
            eventPublisher.publish(payment.pullDomainEvents());
            return new InitiatePaymentResult(payment.getId().getValue(), "BLOCKED", false);
        }

        // ── Step 6: Call NPCI ────────────────────────────────────────────
        // This is an async call — NPCI responds via webhook callback, not synchronously
        // We save the payment in PROCESSING state and NPCI will callback to complete/fail it
        NpciPaymentRequest npciRequest = NpciPaymentRequest.builder()
            .agentTransactionId(payment.getId().getValue())
            .payerVpa(command.payerVpa())
            .payeeVpa(command.payeeVpa())
            .amount(requestedAmount.getAmount())
            .note(command.note())
            .build();

        try {
            npciAdapter.initiatePaymentAsync(npciRequest); // fire-and-forget; callback will complete/fail
        } catch (NpciConnectionException e) {
            // NPCI unreachable — save in PROCESSING, reconciliation will handle it
            // Do NOT mark as FAILED here — it may have succeeded at NPCI
        }

        // ── Step 7: Save & Publish ────────────────────────────────────────
        paymentRepository.save(payment);
        // Domain events go to outbox table in same DB transaction
        // Outbox poller publishes to Kafka after commit
        eventPublisher.publish(payment.pullDomainEvents());

        return new InitiatePaymentResult(payment.getId().getValue(), payment.getStatus().name(), false);
    }

    private UserId resolvePayeeUserId(String payeeVpa) {
        // Look up payee's user ID from VPA registry — returns EXTERNAL if not our user
        return kycService.findUserByVpa(payeeVpa).map(UserKycProfile::userId).orElse(UserId.external());
    }
}
```

### AddMoneyToWalletCommandHandler

```java
@Service
public class AddMoneyToWalletCommandHandler {

    private final WalletRepository walletRepository;
    private final BankAccountACL bankAccountACL;       // for initiating bank debit
    private final DomainEventPublisher eventPublisher;

    @Transactional
    public AddMoneyResult handle(AddMoneyToWalletCommand command) {

        // Idempotency: check Redis for this idempotency key
        // (stored in Redis with 24h TTL after first successful execution)
        if (idempotencyCache.exists(command.clientIdempotencyKey())) {
            return idempotencyCache.get(command.clientIdempotencyKey());
        }

        UserId userId = UserId.of(command.userId());
        Money amount = Money.ofInr(command.amount());

        // Load wallet aggregate
        Wallet wallet = walletRepository.findByOwnerId(userId)
            .orElseThrow(() -> new WalletNotFoundException("No wallet for user: " + userId));

        // Initiate bank debit (async — bank callback will credit wallet)
        // This triggers the AddMoneyToWallet saga (see 07-saga-patterns.md)
        BankDebitRequest debitRequest = new BankDebitRequest(
            command.sourceBankAccountId(), amount, command.clientIdempotencyKey()
        );
        String bankReferenceId = bankAccountACL.initiateDebit(debitRequest);

        // Record that a top-up is in progress
        // Wallet is NOT credited yet — only after bank confirms debit
        TopUpRequest topUpRequest = TopUpRequest.create(
            userId, wallet.getId(), amount, bankReferenceId, command.clientIdempotencyKey()
        );
        topUpRequestRepository.save(topUpRequest);

        AddMoneyResult result = new AddMoneyResult(topUpRequest.getId(), "PENDING");
        idempotencyCache.set(command.clientIdempotencyKey(), result, Duration.ofHours(24));

        return result;
    }
}
```

## NPCI Callback Handler (Application Service)

```java
@Service
public class NpciCallbackHandler {

    private final PaymentRepository paymentRepository;
    private final DomainEventPublisher eventPublisher;

    /**
     * Called when NPCI sends a callback to our webhook endpoint.
     * This is the second half of the UPI payment flow.
     *
     * CRITICAL: this must be idempotent — NPCI may send the same callback multiple times.
     * We use the payment status as the idempotency guard.
     */
    @Transactional
    public void handleCallback(NpciCallbackPayload payload) {

        Payment payment = paymentRepository
            .findById(PaymentId.of(payload.agentTransactionId()))
            .orElseThrow(() -> new PaymentNotFoundException(payload.agentTransactionId()));

        // Guard against duplicate callbacks (idempotency)
        if (payment.getStatus().isTerminal()) {
            log.info("Payment {} already in terminal state {}. Ignoring duplicate callback.",
                payment.getId(), payment.getStatus());
            return;
        }

        if (payload.isSuccess()) {
            payment.complete(payload.npciTransactionId());
        } else {
            payment.fail(payload.errorCode(), payload.errorMessage());
        }

        paymentRepository.save(payment);
        eventPublisher.publish(payment.pullDomainEvents());
    }
}
```

## Query Objects & Handlers (Read Side)

Queries do **not** go through the domain model. They query denormalized read models (projections) directly.

```java
// ─── Query Objects ───

public record GetPaymentDetailsQuery(String paymentId, String requestingUserId) {}

public record GetTransactionHistoryQuery(
    String userId,
    Instant from,
    Instant to,
    int page,
    int pageSize
) {}

public record GetWalletBalanceQuery(String userId) {}

// ─── Read Model (DTOs — not domain objects) ───

public record PaymentDetailsView(
    String paymentId,
    String status,
    String amount,
    String currency,
    String paymentMethod,
    String payerVpa,
    String payeeVpa,
    String payeeDisplayName,
    String initiatedAt,
    String completedAt,
    String failureReason,
    String npciTransactionId
) {}

public record TransactionHistoryItem(
    String transactionId,
    String type,            // SENT, RECEIVED, BILL_PAYMENT, WALLET_TOPUP
    String amount,
    String counterpartyName,
    String counterpartyIdentifier,
    String status,
    String timestamp,
    String referenceId
) {}

public record WalletBalanceView(
    String walletId,
    String availableBalance,
    String heldAmount,
    String totalBalance,
    String kycLevel,
    String status,
    String currency
) {}

// ─── Query Handlers (bypass domain, query read store directly) ───

@Service
public class PaymentQueryHandler {

    // Queries the PostgreSQL read replica (or a denormalized projection in Redis/ES)
    private final PaymentReadModelRepository readModel;

    public PaymentDetailsView handle(GetPaymentDetailsQuery query) {
        return readModel.findPaymentById(query.paymentId())
            .map(this::toView)
            .orElseThrow(() -> new PaymentNotFoundException(query.paymentId()));
    }

    /**
     * Transaction history is a denormalized view across Payment, Wallet, and BillPayment BCs.
     * Built by an event-driven projection: each domain event updates the unified timeline.
     *
     * This is the CQRS "read model" — optimized for the transaction history screen,
     * not derived from the domain aggregates on every request.
     */
    public Page<TransactionHistoryItem> handle(GetTransactionHistoryQuery query) {
        return readModel.findTransactionsForUser(
            query.userId(), query.from(), query.to(),
            PageRequest.of(query.page(), query.pageSize())
        ).map(this::toView);
    }
}

@Service
public class WalletQueryHandler {

    private final WalletReadModelRepository readModel;

    public WalletBalanceView handle(GetWalletBalanceQuery query) {
        return readModel.findByOwnerId(query.userId())
            .map(this::toView)
            .orElseThrow(() -> new WalletNotFoundException(query.userId()));
    }
}
```

## Application Layer Design Rules

| Rule | Rationale |
|---|---|
| **No domain logic in application services** | Logic in domain objects is testable without Spring/JPA; logic in application services requires full context |
| **One command = one transaction** | Keeps transactions short; no distributed transactions in a single command handler |
| **Idempotency checked first** | Before any domain model loaded or business logic executed |
| **Domain events published after commit** | Outbox pattern ensures events aren't lost if publish fails |
| **Read side queries read models, not aggregates** | Avoids loading heavy aggregates just to project a DTO |
| **No lazy loading** | All required data loaded explicitly (no Hibernate lazy proxies crossing service boundaries) |

## Transaction Boundary Diagram

```
@Transactional boundary:
  ┌───────────────────────────────────────────────────────────────┐
  │  1. Idempotency check                                          │
  │  2. Load aggregates (paymentRepository.findBy...)             │
  │  3. Call domain methods (payment.complete(), wallet.debit())  │
  │  4. Save aggregates (paymentRepository.save())                │
  │  5. Write domain events to outbox table                       │
  └───────────────────────────────────────────────────────────────┘
          ↓  Transaction commits
  ┌──────────────────────────────┐
  │  Outbox poller (separate     │
  │  thread) reads events from   │
  │  outbox table, publishes     │
  │  to Kafka, marks as sent     │
  └──────────────────────────────┘
```

This two-phase approach (transactional save + async publish) ensures:
- Events are never published for transactions that rolled back
- Events are never lost even if Kafka is temporarily unavailable
- Exactly-once delivery semantics at the publisher side (consumer-side deduplication handled separately)
